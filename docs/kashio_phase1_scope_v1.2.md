## Phase1 ゴールと前提

### 最優先ゴール

✅ 現場の打刻をデジタル化する（紙・Excelからの脱却）

### 開発方針

- 開発期間: 3〜4ヶ月（じっくり作る）
- 導入順序: まず1店舗で試験運用 → 問題なければ3店舗展開
- 技術方式: Webアプリ（ブラウザで動く・インストール不要）
- 対応デバイス: スタッフのスマホ（メイン）＋既存のiPad（サブ）

### 設計の基本原則

- DBは最初から3店舗対応設計。運用は1店舗から段階的に拡大
- 打刻品質を最優先。月次出力は最低限のシンプルな形式で
- 運用でカバーできる機能は後回し。開発を打刻品質に集中させる

### 多言語対応方針

- Phase1は日本語のみ
- 将来の英語対応に備えて、表示文言は言語ファイルとして外出しする設計にする
- 英語対応が必要になったタイミングで言語ファイルを追加するだけで対応できる形にする

## 機能A: 打刻機能（確定）

### 打刻方式

- **メイン:** スタッフのスマホでQRコードを読み取って打刻
- **サブ:** 既存iPadのブラウザから打刻（QR読み取り不可時の緊急フォールバック）
- **QRコード:** 店舗ごとに1枚印刷して掲示（静的URL、貼り替え不要）

### 認証方式

- メールアドレス + パスワード（全ユーザー共通）
- パスワードリセット機能あり（メール送信方式）
- 店長が管理画面からスタッフの仮パスワードを発行・リセット可能

### スマホ打刻の仕様

- 店舗特定: QRコード読み取りで自動特定
- 従業員特定: ログイン済みのため自動特定（名前タップ不要）
- 打刻時刻: サーバ時刻基準（端末時計に依存しない）
- GPS記録: 打刻時に位置情報を裏で記録（警告・エラーは出さない）
- 複数店舗兼務者: 両方の店舗のQRで打刻可能

**完了画面:**
- 完了メッセージ表示
- 打刻種別・時刻・店舗名の表示
- OKボタン（押すと閉じる）

### iPad打刻の仕様

**認証方式:** 管理者アカウントで常時ログイン済み。スタッフは名前をタップするだけ

**画面の流れ:**
1. 名前選択画面（スタッフ一覧が常時表示）
2. 確認画面（打刻前文言を表示）
3. 出勤 or 退勤の選択
4. 完了表示（数秒後に自動で①に戻る）

**打刻前の表示文言:**
> 本人の打刻ですか？　打刻の際に名前の選択ミスが稀に発生していますので、打刻前に今一度名前を確認してください。

**打刻後の表示文言:**
> 打刻を記録しました。勤務実態と異なる打刻が確認された場合、勤務記録を修正し、会社から確認・指導を行うことがあります。

**打刻忘れ時の表示:**
> 打刻を忘れた場合は、本人以外が代わりに打刻せず、店長または会社に報告してください。

**演出・カメラ:** なし（カメラ権限取得なし・撮影・保存なし）  
**ログ:** 打刻時刻・店舗・端末・操作内容を記録

**iPad打刻の設計経緯:**  
当初、他人打刻への心理的抑止としてシャッター音・フラッシュ演出（実際には撮影しない）の実装を検討した。法務確認の結果、「撮影している」と告知せず「確認音・打刻確認演出」と位置付ける案も検討したが、最終的にはシンプルな確認文言方式を採用。理由は以下の通り。
- 嘘をつかずに抑止力を持たせる方が労使信頼関係の観点から望ましい
- 不正打刻が疑われた場合に論点がずれるリスクを避ける
- 打刻時刻・店舗・端末情報・操作ログの記録で実態管理は十分

### スタッフ向け打刻履歴画面

- 当月の総勤務時間を画面上部に表示
- 月の切り替えで1年分（12ヶ月前まで）遡れる
- 表示項目: 日付・出勤時刻・退勤時刻・勤務時間
- 全所属店舗分の履歴を一覧で確認可能

## 労働時間の計算ルール（確定）

### 基本ルール

| 項目 | 確定内容 |
|---|---|
| 打刻種別 | 出勤・退勤の2打刻のみ（休憩打刻なし） |
| 記録単位 | 分単位の実打刻をそのまま保存 |
| 丸め処理 | システムは行わない（社労士が15分単位で処理） |
| 休憩差し引き | なし（勤務時間は最大6時間程度のため） |
| 営業日区切り | 05:00〜翌04:59を1営業日とする |
| 深夜時間帯 | 22:00〜翌04:59（設定変更可能） |
| 深夜割増率 | 25%（設定変更可能・有効期間付き） |
| 打刻時刻基準 | サーバ時刻基準（端末時計に依存しない） |
| 日またぎ処理 | 05:00またぎは未退勤エラーとして検知・手動修正 |
| 勤務時間計算 | 退勤時刻 − 出勤時刻（分単位） |
| 深夜時間計算 | 22:00〜翌04:59に該当する時間数（分単位） |
| 給与計算 | システムは行わない（社労士が担当） |

### 計算例（出勤 21:00 / 退勤 翌02:00）

- 勤務時間合計: 5時間00分
- 通常時間（21:00〜22:00）: 1時間00分
- 深夜時間（22:00〜翌02:00）: 4時間00分
- → システムはこの数字を記録するだけ。給与計算・割増計算は社労士が行う

### システム設定（オーナーのみ変更可）

| 設定項目 | 初期値 | 変更・備考 |
|---|---|---|
| 給与締め日 | 毎月末日 | 変更可能 |
| 給与支払日 | 翌月20日 | 変更可能 |
| 深夜割増率 | 25% | 変更可能・有効期間付き |
| 深夜時間帯開始 | 22:00 | 変更可能・有効期間付き |
| 深夜時間帯終了 | 翌05:00 | 変更可能・有効期間付き |

> 設定変更後は変更日以降の計算に適用。過去データは変更前の設定値で計算される（有効期間付き管理）。

## 機能C: 勤怠一覧・修正画面 + 速報値（確定）

### ビュー構成

| ビュー名 | 店長 | オーナー | 社労士 |
|---|---|---|---|
| 日別ビュー | 担当店舗のみ | 全店舗 | 全店舗 |
| 人別ビュー | 担当店舗のみ | 全店舗 | 全店舗 |
| 横断ビュー（人別×店舗別） | ❌ | 全店舗 | 全店舗 |
| 人件費速報（従業員明細） | 担当店舗のみ | 全店舗 | ❌ |
| 人件費速報（全店舗合計） | ❌ | ✅ | ❌ |

### 日別ビューの仕様

- 日付の切り替え（前日・翌日ボタン）
- 未退勤エラーのスタッフを一覧の上部に引き上げて赤くエラー表示
- 表示項目: 従業員名・出勤時刻・退勤時刻・勤務時間・深夜時間・ステータス
- 各行に修正画面へのボタン
- 店長は担当店舗のみ（店舗切り替えなし）
- オーナーは店舗切り替えで全店舗を確認可能
- 社労士は閲覧のみ（修正ボタンは表示しない）

### 人別ビューの仕様

- 対象従業員をドロップダウンで選択
- 月の切り替え（前月・翌月ボタン）
- 表示項目: 日付・店舗名・出勤時刻・退勤時刻・勤務時間・深夜時間・ステータス
- 兼務スタッフは同じ日に複数行表示（店舗ごとに1行）
- 月次合計（画面下部）: 店舗ごとの通常時間合計・深夜時間合計＋全店舗の総勤務時間合計

### 横断ビューの仕様

- 対象月の切り替え
- 全従業員×全店舗の月次集計を一覧表示
- 表示項目: 従業員名・店舗別通常h・深夜h・時給・深夜時給・給与小計・交通費・給与合計
- 最終行に全員の合計行
- 月次締めのステータス（open / store_closed / approved）を同一画面で確認可能
- 店舗締めボタン・最終承認ボタン・ロック解除ボタンを同一画面に配置
- 最終承認後はCSV出力ボタンが有効になる

### 未退勤エラーの表示

- 管理画面上部に常時バナー表示＋件数を赤バッジで表示
- クリックするとエラー一覧画面に遷移
- エラー一覧から直接修正画面へ遷移できる
- 社労士には表示しない

### 人件費速報の計算

- 計算式: 通常時間 × 時給 ＋ 深夜時間 × 時給 × 深夜割増率
- 交通費・社会保険・所得税は含まない
- 画面上に「速報値」と明記し、確定値との混同を防ぐ
- 月の途中は「当月の今日時点まで」の集計を表示
- 締め後は「確定値」として表示を切り替える
- 全店舗合計画面は予算比較機能なし（Phase1はシンプルに金額の集計のみ）

### 修正ルール

**修正できる項目と権限:**

| 操作 | 店長 | オーナー |
|---|---|---|
| 出勤時刻・退勤時刻の修正 | 担当店舗 | 全店舗 |
| 店舗・従業員の変更 | ❌ | 従業員マスタから実施 |
| レコード削除 | ❌ | ✅（2段階確認あり） |
| サーバ記録時刻の変更 | ❌ | ❌（証跡として永久保存） |

**レコード削除の2段階確認:**
1. 削除ボタンを押す
2. 「削除しますがよろしいですか？」の確認ダイアログ
3. 「この操作は後戻りできません。実行しますか？」の2段階目の確認ダイアログ
4. 確定後に削除実行

**修正履歴の記録:**
- 修正者・修正日時・修正前の値・修正後の値・修正理由をすべて記録
- 修正理由は選択式（必須）＋自由入力
- 選択肢: 本人が打刻を忘れた / システムエラー / 誤打刻 / その他

### 月次締めワークフロー

| ステップ | 担当 | 内容 |
|---|---|---|
| STEP1 | 店長 | 当月データを確認（未退勤エラーゼロ・異常値なし） |
| STEP1.5 | 店長 | 当月の交通費を従業員ごとに入力 |
| STEP2 | 店長 | 「店舗締め」を実行 → 以降店長による修正不可、オーナーに通知 |
| STEP3 | オーナー | 横断ビューで全店舗を確認（必要に応じてロック解除・修正） |
| STEP4 | オーナー | 「最終承認」を実行 → 全データ完全ロック・CSV出力解放 |
| STEP5 | オーナー/店長 | CSV出力して社労士へ提出 |

## 機能D: 月次CSV出力（確定）

### 出力タイミング・権限

- 出力タイミング: STEP4「最終承認」完了後のみ
- オーナー: 全店舗
- 店長: 担当店舗のみ
- 社労士: 全店舗（閲覧専用）
- スタッフ: 出力不可

### ファイル名

```
kashio_kintal_YYYY年MM月.csv
例: kashio_kintal_2026年4月.csv
```

### フォーマット

- 1行目: ヘッダー行（列名）
- 2行目以降: 従業員ごとの月次集計（1行＝1従業員）
- 最終行: 合計行
- 勤務のなかった店舗の時間・金額: 0.00 または空白で出力
- 出力履歴機能なし

### 列定義

| # | 列名 | 内容 |
|---|---|---|
| 1 | 従業員ID | システム内のID |
| 2 | 氏名 | 漢字氏名 |
| 3 | 酒店_通常h | 酒店での通常勤務時間（時間・小数） |
| 4 | 酒店_深夜h | 酒店での深夜勤務時間（時間・小数） |
| 5 | 酒店_通常時給 | 酒店での通常時給（円） |
| 6 | 酒店_深夜時給 | 酒店での深夜時給（円・通常×深夜割増率） |
| 7 | 酒店_給与 | 酒店分の給与小計（円） |
| 8 | 食堂_通常h | 食堂での通常勤務時間（時間・小数） |
| 9 | 食堂_深夜h | 食堂での深夜勤務時間（時間・小数） |
| 10 | 食堂_通常時給 | 食堂での通常時給（円） |
| 11 | 食堂_深夜時給 | 食堂での深夜時給（円） |
| 12 | 食堂_給与 | 食堂分の給与小計（円） |
| 13 | どんつき_通常h | どんつきでの通常勤務時間（時間・小数） |
| 14 | どんつき_深夜h | どんつきでの深夜勤務時間（時間・小数） |
| 15 | どんつき_通常時給 | どんつきでの通常時給（円） |
| 16 | どんつき_深夜時給 | どんつきでの深夜時給（円） |
| 17 | どんつき_給与 | どんつき分の給与小計（円） |
| 18 | 交通費 | 当月交通費（円） |
| 19 | 給与合計 | 全店舗給与合計＋交通費（円） |

### 交通費の管理仕様

- 月次締めワークフローのSTEP1.5で入力
- 入力できる人: 店長（担当店舗のみ）・オーナー（全店舗）
- 入力項目: 対象従業員・金額（円）・備考（任意）
- 締め後の修正: オーナーのみ可（ロック解除後）
- 入力がない場合: 0円で出力
- 現在の実態: 変動制（実費精算）、将来的に対象者が増える可能性あり

|---|---|---|
| 1: 在籍中 | 可 | 通常表示 |
| 2: 退職済み（給与計算中） | 不可 | 通常表示（「退職済み」バッジあり） |
| 3: 退職済み（アーカイブ） | 不可 | 「退職済みを表示」チェックで切替 |

**ステータス自動移行のタイミング:**
- ステータス1→2: 退職予定日に自動移行
- ステータス2→3: 給与支払日の翌日 00:00 に自動移行

例: 4月末退職 → 5月1日にステータス2移行 → 5月21日00:00にステータス3移行（支払日が翌月20日の場合）

### 申請フロー（2種類）

**時給変更申請:**
1. 店長が申請作成（対象・変更後時給・適用開始日・理由）
2. オーナーへメール通知
3. オーナーが承認 → 指定日から新時給が有効（履歴に記録）・店長へメール通知
4. オーナーが却下 → 却下理由を入力して店長へメール通知

**退職申請:**
1. 店長が申請作成（対象・退職予定日・退職理由）
2. 退職理由の選択肢: 本人都合 / 契約期間満了 / その他（自由入力）
3. オーナーへメール通知
4. オーナーが承認 → 退職予定日に自動でステータス2移行・店長へメール通知
5. オーナーが却下 → 却下理由を入力して店長へメール通知

### 時給管理の仕組み

- 管理単位: 従業員 × 店舗 × 有効期間
- 昇給・降給があった場合は新しい有効期間を追加（過去データは旧時給で計算）
- 時給変更履歴はすべて保存（申請・承認者・変更日も記録）

### その他

- 退職者の扱い: 論理削除（データは完全保存・法令上の保存義務対応）
- 初期データ登録: 英理究さんがDB直接投入（システム導入時の1回限り）
- 以降の追加・編集: 管理画面から操作
