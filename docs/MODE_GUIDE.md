## 1. このガイドについて

このテンプレートでは、Copilot Chat（Agent モード）で **特定のフレーズを伝えるだけ** で、AI の動き方が切り替わります。

```
あなたの指示  →  Copilot が「どのモードか」を自動判定  →  そのモードで動く
```

つまり、**ボタンやメニューでモードを選ぶ必要はありません。** 話しかける言葉が「スイッチ」になっています。

このガイドでは、3つのモードの使い方を **実際に Copilot に伝える言葉の例** と一緒に解説します。

---

## 2. モードの全体像

### どんなモードがある？

| モード | いつ使う？ | AI がやること |
|--------|-----------|--------------|
| 🚀 **自動実行モード** | 計画に書いてあるタスクを実行したいとき | 実装 → テスト → 監査 → PR 作成まで全自動で実行 |
| 📝 **計画修正モード** | やりたいことを追加・変更したいとき | plan.md の更新、GitHub Issue の作成、Project への登録 |
| 💬 **汎用リクエストモード** | 上の2つ以外のこと全般 | 質問への回答、設定変更、調査、コードの改善など |

### 3つのモードの関係

```
📝 計画修正モード          🚀 自動実行モード          💬 汎用リクエストモード
  「何をやるか決める」  →    「計画を実行する」          「それ以外の相談」

  plan.md に書く        →    plan.md の Next を         「○○を調べて」
  Issue を作る               読んで自動で開発            「○○を直して」
```

**基本的な流れ:**

1. 📝 計画修正モードで「やりたいこと」を計画に追加する
2. 🚀 自動実行モードで計画を「実行」する  ← メインの流れ
3. 💬 汎用リクエストモードで日常的な相談・修正を行う

## 3. 自動実行モード（計画に従い作業を実施）

### これは何？

`docs/plan.md` に書かれている **Next（次にやるタスク）** を AI が自動的に実行するモードです。
コードの実装、テストの作成、品質チェック、PR の作成まで一気通貫で行います。

### 🎯 使うタイミング

- Next に実行したいタスクがある
- 「計画書に書いてあるタスクをやってほしい」と思ったとき

### 📎 トリガーフレーズ（これを Copilot に伝える）

以下のフレーズのどれかを Copilot Chat で入力すると、自動実行モードが起動します。

```
計画に従い作業を実施して
```

```
Next を実行して
```

```
plan.md に従って進めて
```

```
作業を開始して
```

```
タスクを実行して
```

> **💡 ポイント**: どのフレーズを使っても同じ動作になります。一番覚えやすいものを使ってください。

### AI が自動で行うこと（全体の流れ）

```
📎 「計画に従い作業を実施して」
  │
  ▼
① plan.md の Next 先頭タスクを確認する
  │
  ▼
② 作業用ブランチを作成する
  │  （main を汚さないようにする）
  │
  ▼
③ 🛠️ コードを実装する
  │  （Implementer エージェントに任せる）
  │
  ▼
④ 🧪 テストを作成する
  │  （Test Engineer エージェントに任せる）
  │
  ▼
⑤ ⚙️ CI チェックを実行する
  │  （リンター・型チェック・テスト・ポリシーチェック）
  │  ❌ 失敗 → 修正して再実行（最大3回）
  │
  ▼
⑥ 📋 3つの監査を実行する
  │  ├── 仕様監査: 要件に合っているか？
  │  ├── セキュリティ監査: 安全か？
  │  └── 信頼性監査: 壊れにくいか？
  │  ❌ Must 指摘あり → 修正して再監査（最大3回）
  │
  ▼
⑦ コミットして PR を作成する
  │
  ▼
⑧ GitHub CI が通るか確認する
  │  ❌ 失敗 → 修正して再プッシュ（最大3回）
  │
  ▼
⑨ 🤖 Copilot コードレビューに対応する
  │
  ▼
⑩ Release Manager が最終判定する
  │
  ▼
⑪ 📢 「PR ができました。確認してください」と報告

あなたが PR を確認 → マージ（ボタンを押す） → 完了！ 🎉
```

### あなたがやること

| タイミング | あなたの操作 |
|-----------|-------------|
| **開始時** | 「計画に従い作業を実施して」と入力する |
| **実行中** | 待つだけ（途中経過が表示される） |
| **完了時** | PR（プルリクエスト）を確認して「マージ」ボタンを押す |

> **⚠️ 注意**: 自動実行中に別の指示を出すと混乱することがあります。完了報告が来るまで待ってください。

### マージした後にやること

PR をマージしたら、**Copilot に以下のように伝える** と、Issue のクローズや計画の更新を自動で行ってくれます。

> **📎 Copilot への指示**:
>
> ```
> マージ完了。計画に従い次の作業を実施して
> ```

この一言で Copilot は：
1. 完了した Issue を Close する
2. GitHub Projects のステータスを Done に更新する
3. plan.md の Done セクションに記録する
4. 次の Next タスクの実行を開始する

もし「次のタスクは後で」と思う場合は、マージ報告だけ伝えてください：

> ```
> マージ完了
> ```

### 止まるとき（停止条件）

以下のケースでは AI が自動的に作業を停止し、あなたに報告します。

| ケース | 何が起きる？ | あなたの対応 |
|--------|------------|-------------|
| plan.md の Next が空 | 実行するタスクがない | 計画修正モードでタスクを追加する |
| 修正ループが3回を超えた | CI や監査を3回直しても通らない | Copilot に状況を聞いて手動で対応 |
| ポリシー違反が見つかった | 禁止されている操作を検出した | Copilot に状況を聞いて計画を見直す |

## 4. 計画修正モード（計画を変更・追加する）

### これは何？

「こんな機能を作りたい」「予定を変更したい」といった要望を `docs/plan.md` に反映し、対応する GitHub Issue も自動で作成してくれるモードです。

### 🎯 使うタイミング

- 新しくやりたいことが出てきたとき
- 既存の計画を変更したいとき
- Issue を追加したいとき

### 📎 トリガーフレーズ（これを Copilot に伝える）

```
計画を修正して
```

```
計画を見直して
```

```
新しい要件を追加して
```

```
Issue を追加して
```

```
Backlog に追加して
```

```
計画を更新して
```

### AI が自動で行うこと

```
📎 「計画を修正して」
  │
  ▼
① あなたの要望をヒアリングする
  │  「どんな機能を追加したいですか？」
  │
  ▼
② 影響を評価する
  │  （既存のフェーズに追加？新フェーズ？）
  │
  ▼
③ docs/plan.md を更新する
  │  ├── Backlog にタスクを追加
  │  ├── 必要なら Next に昇格
  │  └── ロードマップを調整
  │
  ▼
④ GitHub Issue を作成する
  │
  ▼
⑤ GitHub Project に追加する
  │  （進捗ボードに表示される）
  │
  ▼
⑥ 📢 「計画を更新しました」と報告
```

### 使い方の例

#### 例 1: 新しい機能を追加したい

> **📎 Copilot への指示**:
>
> ```
> 計画を修正して。ユーザー登録機能を追加したい。
> メールアドレスとパスワードでサインアップできるようにしたい。
> ```

Copilot は以下のように対応します：

1. 要件の詳細を確認する質問をしてくることがある
2. plan.md の Backlog にタスクを追加する
3. GitHub Issue を作成する
4. Project ボードに登録する

#### 例 2: 計画の順番を変えたい

> **📎 Copilot への指示**:
>
> ```
> 計画を見直して。Backlog にある「ダッシュボード機能」を Next に昇格させて、
> 「通知機能」より先に実装したい。
> ```

#### 例 3: 既存のタスクに変更を加えたい

> **📎 Copilot への指示**:
>
> ```
> 計画を修正して。Issue #15 のタスク内容を変更したい。
> 対象を「全ユーザー」から「管理者ユーザーのみ」に限定する。
> ```

### あなたがやること

| タイミング | あなたの操作 |
|-----------|-------------|
| **開始時** | トリガーフレーズ ＋ やりたいことを伝える |
| **ヒアリング中** | Copilot の質問に答える |
| **完了時** | plan.md の内容を確認する（GitHub Issues / Project も確認すると尚可） |

## 5. 汎用リクエストモード（その他の依頼）

### これは何？

自動実行モード・計画修正モードのどちらのトリガーフレーズにも当てはまらないすべてのリクエストを処理するモードです。
Copilot が自動的にこのモードで動作するので、**特別なフレーズは不要** です。

### 🎯 使うタイミング

- コードの改善やリファクタリングをしたいとき
- エラーの原因を調べてほしいとき
- 設定を変更してほしいとき
- 何かについて質問したいとき
- ドキュメントを修正したいとき

### 📎 使い方（そのまま話しかけるだけ）

```
このファイルのバグを修正して
```

```
テストが失敗している原因を調べて
```

```
Python のバージョンを 3.12 に上げたい
```

```
docs/architecture.md を最新の設計に合わせて更新して
```

```
CI が失敗しているので直して
```

### コード変更を伴う場合の品質管理

汎用リクエストモードでも、**コードを変更する場合は品質チェックが自動で実行されます**。

```
📎 「この関数のバグを直して」
  │
  ▼
① コードを修正する
  │
  ▼
② 自動で品質チェックを実行 ← ここが重要！
  │  ├── リンター（コードの書き方チェック）
  │  ├── フォーマッター（整形チェック）
  │  ├── 型チェック（型の整合性）
  │  ├── テスト実行
  │  └── ポリシーチェック（禁止パターンの検出）
  │
  ▼
③ 全てパスしたら完了を報告

❌ チェックに失敗 → 自動で修正を試みる
```

### ドキュメントのみの変更

`docs/` フォルダのドキュメントだけを変更する場合は、上記の品質チェックは省略されます。ただし、ドキュメント間の整合性（計画⇔設計⇔要件）は確認されます。

### PR が必要な変更の場合

Copilot が「PR で管理した方がよい」と判断した場合、自動実行モードと同様の品質管理フロー（監査 → PR 作成 → CI 確認 → レビュー対応 → リリース判定）が実行されます。

> **💡 ポイント**: PR が必要かどうかは Copilot が判断します。大きな変更や `src/` の変更は PR 経由になることが多いです。小さな修正やドキュメントの更新は直接コミットされることがあります。

## 6. モードの使い分け早見表

### 「こんなときはこのモード」

| やりたいこと | 使うモード | Copilot に伝える言葉 |
|-------------|-----------|---------------------|
| 計画のタスクを実行したい | 🚀 自動実行 | 「計画に従い作業を実施して」 |
| 新しい機能を計画に追加したい | 📝 計画修正 | 「計画を修正して。○○を追加したい」 |
| 計画の優先順を変えたい | 📝 計画修正 | 「計画を見直して。○○を先にやりたい」 |
| バグを見つけたので直したい | 💬 汎用 | 「このバグを修正して」 |
| コードの意味を知りたい | 💬 汎用 | 「この関数は何をしている？」 |
| 設定を変更したい | 💬 汎用 | 「Python のバージョンを変えたい」 |
| CI のエラーを直したい | 💬 汎用 | 「CI が失敗しているので直して」 |
| 次のタスクを実行したい | 🚀 自動実行 | 「Next を実行して」 |
| Issue を作りたい | 📝 計画修正 | 「Issue を追加して」 |
| ドキュメントを更新したい | 💬 汎用 | 「docs の内容を更新して」 |

### フローチャート：迷ったときの選び方

```
「Copilot に何かやってもらいたい」
  │
  ├── plan.md の Next にあるタスクを実行してほしい？
  │   │
  │   ├── はい → 🚀 自動実行モード
  │   │         「計画に従い作業を実施して」
  │   │
  │   └── いいえ ↓
  │
  ├── やりたいことを計画に追加・変更したい？
  │   │
  │   ├── はい → 📝 計画修正モード
  │   │         「計画を修正して。○○したい」
  │   │
  │   └── いいえ ↓
  │
  └── その他（質問・修正・調査・設定変更など）
      │
      └── → 💬 汎用リクエストモード
            そのまま話しかけるだけ！
```

## 7. 実践例：プロジェクト開発の流れ

実際の開発で3つのモードをどう使い分けるか、具体的なシナリオで説明します。

### シナリオ: 「TODOアプリ」の開発

#### Step 1: 計画を立てる（📝 計画修正モード）

プロジェクトを始めたばかりで、まだ何も計画がない状態です。

> **📎 Copilot への指示**:
>
> ```
> 計画を修正して。以下の機能を作りたい：
> 1. TODO の追加機能
> 2. TODO の一覧表示機能
> 3. TODO の完了チェック機能
> 4. TODO の削除機能
>
> まず「追加」と「一覧表示」を先にやりたい。
> ```

**Copilot が行うこと:**
- plan.md に 4 つのタスクを追加（2つを Next、2つを Backlog に配置）
- GitHub Issues を 4 つ作成
- Project ボードに登録

#### Step 2: 最初のタスクを実行する（🚀 自動実行モード）

> **📎 Copilot への指示**:
>
> ```
> 計画に従い作業を実施して
> ```

**Copilot が行うこと:**
- Next の先頭タスク（TODO の追加機能）を実装
- テストを作成
- 品質チェック → 監査 → PR 作成
- 完了報告：「PR #1 を作成しました。確認してください」

**あなたがやること:**
- PR を確認 → マージボタンを押す

#### Step 3: 途中でバグに気づいた（💬 汎用リクエストモード）

マージ後に動かしてみたら、入力が空のときにエラーが出ることに気づきました。

> **📎 Copilot への指示**:
>
> ```
> TODO の追加で、タイトルが空のときにエラーになるバグがある。
> 空のときはエラーメッセージを表示するようにして。
> ```

**Copilot が行うこと:**
- バグの原因を調査
- コードを修正
- テストも追加
- 品質チェックを実行
- 完了報告

#### Step 4: 次のタスクを実行する（🚀 自動実行モード）

> **📎 Copilot への指示**:
>
> ```
> 次のタスクを実行して
> ```

**Copilot が行うこと:**
- Next の次のタスク（TODO の一覧表示機能）を実装
- 同様に PR 作成まで自動実行

#### Step 5: 急ぎの要件が入った（📝 計画修正モード）

「削除機能」より先に「期限設定機能」が必要になりました。

> **📎 Copilot への指示**:
>
> ```
> 計画を修正して。「TODO の期限設定機能」を新しく追加して、
> 「削除機能」より優先したい。
> ```

**Copilot が行うこと:**
- plan.md を更新（期限設定を Backlog に追加し、削除機能より上に配置）
- GitHub Issue を作成

## 8. カスタムエージェント（@メンション）の使い方

このテンプレートには **7 つの AI エージェント** が組み込まれています。通常は自動実行モードで Orchestrator が自動的に各エージェントに仕事を振り分けますが、特定のエージェントに直接話しかけることもできます。

### エージェント一覧

| エージェント | @メンション名 | 役割 |
|-------------|--------------|------|
| 🎯 **Orchestrator** | `@orchestrator` | 司令塔。他のエージェントに指示を出す |
| 🛠️ **Implementer** | `@implementer` | コードを書く実装担当 |
| 🧪 **Test Engineer** | `@test-engineer` | テストを書く担当 |
| 📋 **Auditor Spec** | `@auditor-spec` | 仕様（要件）に合っているか監査 |
| 🔒 **Auditor Security** | `@auditor-security` | セキュリティの監査 |
| 🔧 **Auditor Reliability** | `@auditor-reliability` | 信頼性・エラーハンドリングの監査 |
| ✅ **Release Manager** | `@release-manager` | リリース可能かの最終判定 |

### @メンションの使い方

Copilot Chat の入力欄で `@` を入力すると、エージェントの候補が表示されます。

> **📎 例: Implementer に直接コードを書いてもらう**:
>
> ```
> @implementer ユーザー登録のバリデーション関数を実装して
> ```

> **📎 例: テストだけ書いてもらう**:
>
> ```
> @test-engineer src/auth/validator.py のユニットテストを書いて
> ```

> **📎 例: セキュリティの観点でコードをチェックしてもらう**:
>
> ```
> @auditor-security src/api/handler.py をセキュリティの観点で監査して
> ```

### いつ @メンションを使う？

| 場面 | 使い方 |
|------|--------|
| **通常の開発** | @メンション不要（Orchestrator が自動で振り分ける） |
| **特定の作業だけやりたい** | 該当するエージェントに @メンション |
| **セキュリティが心配なコード** | `@auditor-security` で直接チェック |
| **テストだけ追加したい** | `@test-engineer` で直接依頼 |

> **💡 ポイント**: 普段は @メンションを使わなくても大丈夫です。「計画に従い作業を実施して」で Orchestrator が適切に判断してくれます。

## 9. よくある質問（FAQ）

### Q1: 3つのモードは手動で切り替えるの？

**いいえ**、自動で判定されます。Copilot が「あなたの言葉」からどのモードかを自動的に判断します。
「計画に従い作業を実施して」と言えば自動実行モード、「計画を修正して」と言えば計画修正モード、それ以外は汎用リクエストモードです。

### Q2: VS Code の Agent / Edit / Ask モードとの関係は？

VS Code の **モード選択**（Agent / Edit / Ask）と、このテンプレートの **3つの動作モード**（自動実行/計画修正/汎用リクエスト）は別物です。

```
VS Code のモード            このテンプレートのモード
┌─────────────┐            ┌─────────────────────┐
│ Agent モード │ の中で  →  │ 🚀 自動実行モード    │
│（推奨）     │ 動く       │ 📝 計画修正モード    │
│             │            │ 💬 汎用リクエストモード│
├─────────────┤            └─────────────────────┘
│ Edit モード │ → テンプレートのモードとは無関係
├─────────────┤
│ Ask モード  │ → テンプレートのモードとは無関係
└─────────────┘
```

**このテンプレートの3つのモードは、VS Code の Agent モードの中で動きます。** Edit モードや Ask モードでは、テンプレートの自動化機能は使えません。

### Q3: 自動実行中に別の指示を出してもいい？

**推奨しません。** 自動実行モードは多くのステップを順番に実行するため、途中で別の指示を出すと混乱することがあります。完了報告を待ってから次の指示を出してください。

### Q4: 自動実行モードで Next が複数ある場合は？

先頭のタスクから1つずつ実行します。1つ完了したら、次のタスクを実行するには再度「計画に従い作業を実施して」と伝えてください。

### Q5: 計画修正モードで間違った計画を作ってしまった場合は？

もう一度「計画を修正して」と伝えて、修正内容を説明してください。Copilot が plan.md を更新し、不要な Issue があればクローズしてくれます。

### Q6: 汎用リクエストモードで大きな変更をすると PR になる？

はい。コードファイル（`src/`、`tests/` など）を変更する場合、Copilot が品質チェックを実行し、必要に応じて PR を作成します。ドキュメントだけの変更では PR は不要と判断されることがあります。

### Q7: 自動実行モードが途中で止まったらどうする？

Copilot が停止理由を報告します。以下を確認してください：

1. **plan.md の Next が空**: 計画修正モードでタスクを追加する
2. **修正ループ超過**: Copilot に状況を聞いて、手動で問題を解決する
3. **ポリシー違反**: 計画自体を見直す必要がある

> **📎 状況を確認したいとき**:
>
> ```
> さっきの自動実行が途中で止まった原因を教えて
> ```

### Q8: マージは自動でされる？

**いいえ。** マージは必ずあなた（人間）が確認してから行います。AI は PR を作成するところまでで、マージボタンを押すのはあなたの判断です。これは安全策として意図的にそうなっています。

### Q9: トリガーフレーズを英語で伝えてもいい？

基本的にこのテンプレートは日本語を前提としていますが、「Execute the plan」「Start the task」のような英語でも動作する場合があります。ただし、確実に動作させるには日本語のトリガーフレーズを使うことを推奨します。

### Q10: テンプレートをアップデートしたらモードの動作は変わる？

テンプレートのアップデートにより、モードの動作が改善されることがあります。アップデート方法は [テンプレートアップデートガイド](UPGRADE_GUIDE.md) を参照してください。

## 10. トラブルシューティング

### 自動実行モードが起動しない

**症状**: 「計画に従い作業を実施して」と伝えたのに、普通の会話が返ってくる。

**原因と対処**:

| 原因 | 対処 |
|------|------|
| Agent モードになっていない | Chat パネル上部のドロップダウンを **Agent** に変更する |
| plan.md の Next が空 | 計画修正モードでタスクを追加する |
| 新しいチャットではない | Chat パネル上部の **＋** ボタンで新しいチャットを開始する |
| Copilot の応答上限に達した | しばらく待ってから再度試す |

### 計画修正モードで Issue が作成されない

**症状**: plan.md は更新されたが、GitHub Issue が作られていない。

**原因と対処**:

| 原因 | 対処 |
|------|------|
| GitHub CLI にログインしていない | ターミナルで `gh auth login` を実行する |
| 権限が不足している | リポジトリの Write 権限があるか確認する |
| GitHub API の一時的なエラー | もう一度「Issue を追加して」と伝える |

> **📎 GitHub CLI のログイン状態を確認する**:
>
> ```
> gh auth status を確認して
> ```

### 品質チェックが何度も失敗する

**症状**: 汎用リクエストモードで修正したが、CI チェックが通らない。

**対処**:

1. Copilot にエラー内容を聞く：「CI のエラーの原因を教えて」
2. エラーが理解できなくても、Copilot に「このエラーを修正して」と伝える
3. 3回試しても解決しない場合は、エラーの詳細を添えて相談する

> **📎 CI エラーの確認**:
>
> ```
> さっきの品質チェックで失敗した項目と原因を教えて
> ```

### エージェントが応答しない

**症状**: `@orchestrator` と入力しても反応がない。

**対処**:

| 確認事項 | 方法 |
|---------|------|
| エージェント定義ファイルの存在 | `.github/agents/` フォルダにファイルがあるか確認 |
| VS Code の再起動 | VS Code を閉じて再度開く |
| Copilot 拡張機能の更新 | 拡張機能パネルでアップデートがないか確認 |
| Agent モードであること | モードが Agent になっているか確認 |

### 「ポリシー違反」で停止した

**症状**: 自動実行中に「ポリシー違反が検出されました」と表示されて停止した。

**対処**:

1. どのポリシーに違反したか確認する（Copilot が教えてくれる）
2. 計画自体がポリシーに反していないか見直す
3. 必要なら計画修正モードで計画を変更する

> **📎 ポリシーの確認**:
>
> ```
> docs/policies.md の内容を教えて
> ```

---

> **最後に**: モードの使い分けに迷ったら、**とりあえず普通に話しかけてください**（汎用リクエストモード）。
> Copilot がうまく対応してくれます。自動実行モードを使いたくなったら「計画に従い作業を実施して」、
> 計画を変えたくなったら「計画を修正して」—— これだけ覚えれば十分です。
